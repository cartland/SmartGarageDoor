# Play Store assets

Uploading to the Play Store is a **manual process** — there is no automation
wired up (the release workflow only ships the AAB + `whatsnew/`). This skill is
the procedure for refreshing the **store listing** images.

## The two places assets live

| Location | What | Committed? |
| --- | --- | --- |
| `AndroidGarage/screenshots/store/` | **Staging** — everything the generators produce | No (gitignored) |
| `AndroidGarage/distribution/playstore/` | **Curated** — the exact set we keep in sync with the live store | Yes |

Generators **only write to staging**. `distribution/` is updated by hand: you
copy the images you actually want, open a PR, then upload them in Play Console.
Never point a generator at `distribution/`.

## Generators (run from repo root)

```bash
# Icon (512) + feature graphic -> staging.  Also regenerates the in-app launcher
# mipmaps in androidApp/.../res/ (those ARE committed app code — see note below).
bash AndroidGarage/distribution/playstore/generate.sh

# Phone + 7" + 10" tablet screenshots -> staging.  Composes committed reference
# PNGs (no rendering), so it works even though local Layoutlib render is blank.
python3 scripts/generate-store-screenshots.py
```

Both write under `AndroidGarage/screenshots/store/`:

```
screenshots/store/
  icon-512.png
  feature-graphic-1024x500.png
  phoneScreenshots/        01..06  (1544x2744, 9:16)
  sevenInchScreenshots/    01..04  (~3700x2100 / 3584x2016, 16:9)
  tenInchScreenshots/      01..04  (~3734x2100, 16:9)
```

All output is Play-compliant: PNG, < 8 MB, exact 16:9 / 9:16, sides within range.

## Updating the store — step by step

1. **Regenerate** with the two commands above (do this after any icon/screen UI
   change, and refresh the source reference PNGs first if the UI changed — see
   _Refreshing screenshot sources_).
2. **Review** the staged images in `AndroidGarage/screenshots/store/`.
3. **Copy the ones you want** into `distribution/playstore/`:

   | Copy from staging | Copy to distribution | Play Console field |
   | --- | --- | --- |
   | `icon-512.png` | `distribution/playstore/icon-512.png` | Main store listing → App icon |
   | `feature-graphic-1024x500.png` | `distribution/playstore/feature-graphic-1024x500.png` | Main store listing → Feature graphic |
   | `phoneScreenshots/*.png` (2–8) | `distribution/playstore/screenshots/phoneScreenshots/` | Phone screenshots |
   | `sevenInchScreenshots/*.png` (≤8) | `distribution/playstore/screenshots/sevenInchScreenshots/` | 7-inch tablet screenshots |
   | `tenInchScreenshots/*.png` (≤8) | `distribution/playstore/screenshots/tenInchScreenshots/` | 10-inch tablet screenshots |

   You don't have to copy all of them — pick the subset you want live. Phone
   needs 2–8; tablets are ≤8 and only required if targeting large screens.
4. **Open a PR** with the new `distribution/playstore/` images. This is the
   record of what's live in the store.
5. **After merge, upload manually** in Play Console (Store presence → Main store
   listing), one field at a time per the table above. Save the listing.

## Important distinctions

- **Launcher icon ≠ store icon.** `generate.sh` writes the in-app launcher
  mipmaps into `androidApp/.../res/mipmap-*` — that's app code, it ships in the
  AAB automatically on the next `release-android`, and is NOT a manual upload.
  The 512 `icon-512.png` is the separate *store listing* icon and IS a manual
  upload. Both come from the same `src/icon.svg`.
- **Aspect ratio.** Play's hard limit is "longest side ≤ 2× shortest side"; the
  generators pad to an exact 16:9 / 9:16 to stay safely inside that and match
  the nominal requirement.

## Refreshing screenshot sources

Phone screenshots come from `AndroidGarage/screenshots/framed/` and tablet
screenshots from the committed AGP reference PNGs
(`android-screenshot-tests/.../reference/.../ScreensScreenshotTestKt/`). Those
are regenerated by the normal screenshot pipeline
(`./scripts/generate-android-screenshots.sh`), which **renders blank on this
Mac** (see root `CLAUDE.md`). So a true UI refresh of the *sources* needs a
working environment / CI; `generate-store-screenshots.py` itself only re-composes
whatever sources are currently committed.

## Optional: automate the upload later

If hand-uploading becomes tedious, the upload step can be automated with
`fastlane supply` or the Play Developer API `edits.images.upload`, reusing the
existing `GOOGLE_PLAY_SERVICE_ACCOUNT_JSON` secret (it would need the "Edit store
listing" permission added). Gate it behind `workflow_dispatch` so it never fires
on every release. Not built yet — intentionally manual.
