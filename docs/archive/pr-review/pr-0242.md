# PR #242: refactor: AppSettingsViewModel uses AppSettingsUseCase

**Goal:** Phase 43 second step. `AppSettingsViewModel` no longer depends on `AppSettingsRepository` directly. New `AppSettingsUseCase` façade with per-setting observe/update methods (fcmDoorTopic, profileUserCardExpanded, profileLogCardExpanded, profileAppCardExpanded). One UseCase with 8 methods is cleaner than 8 separate single-method UseCases for this case.

**Files:** `AndroidGarage/usecase/.../AppSettingsViewModel.kt`, `AppSettingsUseCase.kt` (new), `DefaultAppSettingsViewModelTest.kt`, `AppComponent.kt`.

**Rationale:** One façade UseCase per related-setting group is the right granularity here — 8 trivial single-method UseCases would have been worse ergonomics with no real benefit.

**Still-current lesson:** UseCases can be a *façade* for related repo methods; splitting one per method is a dogma that hurts readability when all methods serve the same screen.
