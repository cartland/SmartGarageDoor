# PR #256: fix: Snooze network errors actually propagate to the UI

**Goal:** `SnoozeAction.Failed.NetworkError` was dead code — repo swallowed all network errors and returned Unit, so UseCase always returned `AppResult.Success` and UI never saw the network error state. `SnoozeRepository.snoozeNotifications` now returns Boolean. `NetworkSnoozeRepository` propagates HTTP errors and connection failures as false. `SnoozeNotificationsUseCase` translates false to `AppResult.Error(NetworkFailed)`. New `ActionError.NetworkFailed` variant. ViewModel maps to `SnoozeAction.Failed.NetworkError`. 4 new tests.

**Files:** `AndroidGarage/data/.../NetworkSnoozeRepository.kt`, `PushRepositoryTest.kt`, `domain/.../ActionError.kt`, `SnoozeRepository.kt`, `test-common/.../FakeSnoozeRepository.kt`, `usecase/.../SnoozeNotificationsUseCase.kt`, `RemoteButtonViewModel.kt`, `RemoteButtonViewModelTest.kt`.

**Rationale:** Dead `NetworkError` branch → real propagation is the right diagnosis, and `ActionError.NetworkFailed` still lives. The `Boolean` return shape was later upgraded to `AppResult<SnoozeState, ActionError>` (#354), but the move to "repo conveys failure explicitly" is still the model.

**Still-current lesson:** "Swallow errors and return Unit" looks tidy in a repo but kills UI error states downstream — every network-bearing repo call needs a return value that carries success vs. typed failure, not a boolean + `null` chorus.
