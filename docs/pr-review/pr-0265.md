# PR #265: fix: Button timeout regression — start timer from HTTP request

**Goal:** `ButtonStateMachine` was starting the 10s network timeout on confirm tap, but HTTP doesn't begin until after token refresh + config fetch. This caused false `SendingTimeout` while server actually received the request and door opened. Moved timeout start to when `PushStatus.SENDING` arrives. New test verifies state machine stays in `Sending` indefinitely before `PushStatus.SENDING`.

**Notes:** This PushStatus.SENDING dependency caused the PR #276 stuck-in-Sending bug when UseCase fails and never emits SENDING.

**Files:** `AndroidGarage/usecase/src/commonMain/kotlin/com/chriscartland/garage/usecase/ButtonStateMachine.kt`, `ButtonStateMachineTest.kt`.
