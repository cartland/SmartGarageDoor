# PR #229: feat: Snooze UX — SnoozeState + SnoozeAction with loading and error states

**Goal:** Replace `SnoozeRequestStatus` enum with two independent sealed types. `SnoozeState` (Loading/NotSnoozing/Snoozing) — always visible, never lost during saves. `SnoozeAction` (Idle/Sending/Succeeded/Failed.*) — overlays on state. New `FetchSnoozeStatusUseCase` + `ObserveSnoozeStateUseCase`. Loading spinner replaces Save button during Sending. Succeeded shows optimistic snooze time, auto-resets to Idle after 10s. Failed subtypes: NotAuthenticated, MissingData, NetworkError. UI uses exhaustive `when` on both sealed types.

**Files:** `AndroidGarage/domain/.../SnoozeNotificationsModel.kt`, `data/.../NetworkSnoozeRepository.kt`, `usecase/.../RemoteButtonViewModel.kt`, `FetchSnoozeStatusUseCase.kt` (new), `ObserveSnoozeStateUseCase.kt` (new), `androidApp/ui/SnoozeNotificationCard.kt`, `ProfileContent.kt`, `test-common/.../FakeSnoozeRepository.kt`, `presentation-model/.../ProfileScreenState.kt`, `SnoozeRepository.kt`, `AppComponent.kt`, test files.

**Rationale:** Separating state (always visible) from action (overlay, auto-resets) is the right shape — UX never loses the thing the user's looking at while transient "Sending/Succeeded/Failed" feedback comes and goes. Pattern still in use and generalized.

**Still-current lesson:** "Current state" and "action-in-progress/outcome" are two *independent* streams — modeling them as one enum forces UI to conflate them. Split into `State` + `Action` sealed types, render both, and let Action auto-reset after a display window.
