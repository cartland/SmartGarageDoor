# PR #237: feat: Add unified ButtonStateMachine + RemoteButtonState (PR A of 3)

**Goal:** PR A of button state machine refactor. Problem: UI tap-to-confirm logic lives inside Composable using real `delay()` (untestable), coordinates with separate network state machine via callbacks, tests can't catch timing bugs. This PR adds new state machine without touching existing code. New `RemoteButtonState` sealed interface (9 states: Ready/Arming/Armed/NotConfirmed/Sending/Sent/Received/SendingTimeout/SentTimeout). New `ButtonStateMachine` in usecase/commonMain: single Channel consumer serializes all events through one coroutine (atomic, no races), virtual time injected dispatcher, `drop(1)` skips StateFlow initial replay, optimistic Sending on confirm tap, pure (no UseCase/repo deps), `onSubmit` callback. 22 tests with virtual time.

**Files:** `AndroidGarage/domain/.../RemoteButtonModel.kt`, `usecase/.../ButtonStateMachine.kt` (new, 250 lines), `ButtonStateMachineTest.kt` (new, 391 lines).

**Rationale:** Channel-based serialized state machine with injected dispatcher + `drop(1)` to skip StateFlow replay is the foundational design that survived variant renames (#266) and state-ownership migrations. 22 virtual-time tests anchor it.

**Still-current lesson:** For a state machine you want to test with virtual time, three pieces fit together: a single `Channel` consumer (serializes races), an injected `TestDispatcher` (no real `delay`), and `drop(1)` on the state-flow subscription (avoids replaying the initial state as a tick event).
