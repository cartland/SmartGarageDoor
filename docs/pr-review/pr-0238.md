# PR #238: refactor: Wire ButtonStateMachine + stateless Composable (PR B of 3)

**Goal:** PR B of the button state machine refactor. `DefaultRemoteButtonViewModel` uses `ButtonStateMachine` (from PR #237) instead of old `RemoteButtonStateMachine`. Exposes `buttonState: StateFlow<RemoteButtonState>`. New API: `onButtonTap()`, `resetButton()`. `RemoteButtonContent` becomes a stateless renderer — takes `state` + `onTap`, removes ~80 lines of coroutineScope/delay/remember tap-to-confirm logic. Exhaustive `when` over `RemoteButtonState`. Deletes old `RemoteButtonStateMachine`, its test, `RequestStatus` enum, UI-only `enum class RemoteButtonState`. Tests rewritten to focus on wiring (SM ↔ UseCase).

**Files:** `AndroidGarage/androidApp/.../ui/HomeContent.kt`, `RemoteButtonContent.kt`, `domain/.../RemoteButtonModel.kt`, `presentation-model/.../HomeScreenState.kt`, `usecase/.../RemoteButtonViewModel.kt`, delete `RemoteButtonStateMachine.kt` + test, `RemoteButtonViewModelTest.kt`.

**Rationale:** Turning the Composable into a stateless `state + onTap` renderer — moving ~80 lines of coroutine/delay/remember logic into the VM-owned state machine — is the pattern that makes UI tests trivial. Exhaustive `when` over `RemoteButtonState` is the enforcement.

**Still-current lesson:** A Composable with `delay()`, `coroutineScope {}`, or `remember { mutableStateOf(...) }` for business transitions is hiding a state machine. Extract it, make the Composable stateless.
