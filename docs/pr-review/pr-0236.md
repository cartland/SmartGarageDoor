# PR #236: fix: Snooze Loading stuck + 'Do not snooze' shows wrong time

**Goal:** Two snooze UX bugs. (1) Loading stuck forever — `fetchSnoozeStatus()` had two early-return paths that never cleared Loading (`serverConfig == null`, feature flag disabled); now clears in both cases. (2) "Do not snooze" showed "Snoozing until <current time>" (optimistic time = currentTime + 0h = current time). Refactor `SnoozeAction.Succeeded` into sealed type: `Cleared` → "Notifications resumed", `Set(until)` → "Snoozing until X". New `SnoozeNotificationCardClearedPreview` screenshot.

**Files:** `AndroidGarage/androidApp/.../SnoozeNotificationCard.kt`, `domain/.../SnoozeNotificationsModel.kt`, `data/.../NetworkSnoozeRepository.kt`, `PushRepositoryTest.kt`, `usecase/.../RemoteButtonViewModel.kt`, `RemoteButtonViewModelTest.kt`, screenshot PNGs.

**Rationale:** `SnoozeAction.Succeeded.Cleared` vs `Set(until)` is the right sealed split — a 0-duration "do not snooze" is a semantically different outcome from "snoozed until X", and collapsing them into one success with `until=now` was the bug source.

**Still-current lesson:** "Succeeded" is not always one shape — if a command can succeed with multiple semantically distinct outcomes, make them different sealed variants so the UI is forced to render each one correctly.
