# Android Changelog

Internal release history. For Play Store "What's New" text, see `distribution/whatsnew/`.

## Versioning

- **Major (X.0.0)** — App rewrite or a change in the core experience so significant the previous version feels like a different product.
- **Minor (X.Y.0)** — A new user-facing feature or capability (something a user couldn't do before), **or** the removal of a user-facing feature.
- **Patch (X.Y.Z)** — Bug fixes, UI polish, performance, refactors. No new capability.

Every version gets an entry in this file (internal history). Play Store `distribution/whatsnew/` gets a line per minor/major — patches roll up into the next minor's line, or get a combined line if promoted to production on their own.

## 2.16.33
- **`checkPreviewTime` build-time check** ported from battery-butler. Two-pass BFS that ensures `@Preview` Composables never transitively call a Composable whose default parameter value uses `Clock.System.now()` / `Instant.now()` / `System.currentTimeMillis()` etc. without explicitly passing the time parameter.
  - **Why this matters**: `@Preview` and screenshot tests render at preview-time. A default like `now: Instant = Instant.now()` reads wall-clock at render — screenshots flap, golden-image tests get false-positive diffs, framed README screenshot drifts on every regen.
  - **Pass 1**: collect "time-sensitive" Composables (those with a `now()`-defaulted parameter).
  - **Pass 2**: BFS from each `@Preview` Composable through the call chain (depth-limited at 6). For every call to a time-sensitive Composable, verify the time-sensitive parameter was explicitly passed.
  - **Codebase is clean at port time** — `rememberDurationSince` (the only Composable that calls `Instant.now()`) reads wall-clock from inside `LaunchedEffect` (not a default param) and short-circuits via `LocalInspectionMode` to a fixed value in previews. Zero violations; this prevents drift.
  - **Suppression**: add `// @PreviewTimeExempt` on the call line if intentional.
  - Wired into `validate.sh`. Adapted to use SmartGarageDoor's `sourceDirs: List<String>` convention (vs. battery-butler's `project.rootDir.walk()`).

## 2.16.31
- **Default Nav rail top padding is now derived, not hardcoded.** Pre-2.16.31 the value `8` appeared as a literal in 8 places (`DataStoreAppSettings`, `FakeAppSettingsRepository`, `ProfileViewModel` initial state, 5 preview call sites). The relationship between `8` and `Spacing.ListContentPadding.top` (16 dp) was implicit — a future bump of `Spacing.ListContentPadding.top` (e.g. for a more spacious section rhythm) would silently break rail-vs-content alignment until someone smoke-tested on hardware. Now the default is **derived** from a single named relationship: `DEFAULT_TOP_PADDING_DP = BODY_CONTENT_TOP_DP − RAIL_INTRINSIC_PILL_OFFSET_DP`.
  - **User-facing**: no behavior change. Default is still 8 dp; users can still override via Settings → Developer → Nav rail.
  - **Internal: new domain object `NavigationRailLayout`** (in `:domain/.../model/NavigationRailLayout.kt`) with three named constants — `BODY_CONTENT_TOP_DP = 16`, `RAIL_INTRINSIC_PILL_OFFSET_DP = 8`, `DEFAULT_TOP_PADDING_DP = 8` (derived). Class KDoc explains the math (body-content-top vs rail-internal-padding) and the M3-internals drift risk (the `8` sum captures `NavigationRailVerticalPadding` 4 dp + `NavigationRailItem`'s own pre-pill padding 4 dp, neither of which is exposed by `androidx.compose.material3`).
  - **Internal: 3 source-of-truth defaults** updated to use the derived constant: `DataStoreAppSettings.navigationRailTopPaddingDp`, `FakeAppSettingsRepository.navigationRailTopPaddingDp`, `ProfileViewModel._navigationRailTopPaddingDp`.
  - **Internal: 7 preview call sites** in `:androidApp` updated from `navigationRailTopPaddingDp = 8` → `NavigationRailLayout.DEFAULT_TOP_PADDING_DP`.
  - **Internal: build-time pin** — new `SpacingTest.navRailBodyContentTopMatchesListContentPaddingTop` asserts `Spacing.ListContentPadding.top.value == NavigationRailLayout.BODY_CONTENT_TOP_DP`. If a future PR bumps `Spacing.ListContentPadding.top` (e.g. 16 → 24) without also updating the domain constant, the test fails the build with a message pointing at `NavigationRailLayout` KDoc.

## 2.16.30
- **Spacing rule audit fixes — bottom sheets and history.** Follow-up to 2.16.29 ("container owns the gap"). An audit found 5 more violations of the same rule across bottom sheets and the door-history screen. All fixed in this release; no children claim a vertical gap above or below themselves.
  - **Snooze sheet**: dropped `Modifier.padding(bottom = 8.dp)` from the title `Text`. The parent Column already provides 8 dp via `spacedBy(8.dp)`; the title was double-padding to **16 dp** while subsequent radio rows had **8 dp** — visible inconsistency.
  - **Version sheet**: restructured the flat Column with ad-hoc `Spacer(8.dp)` and `Spacer(16.dp)` siblings into two nested Columns — header cluster (Icon + Title, `spacedBy(16.dp)`) and field rows (`spacedBy(8.dp)`) — joined by an outer Column's `spacedBy(24.dp)`. Visual rhythm preserved (icon→title 16 dp, title→fields 24 dp, field→field 8 dp); `Spacer` import dropped. Inner `VersionFieldRow` Card body's `Spacer(2.dp)` between label and value also replaced by `verticalArrangement = spacedBy(2.dp)` on the wrapping Column.
  - **Account sheet**: same restructure — outer Column `spacedBy(24.dp)`, identity sub-cluster (name + email) with `spacedBy(4.dp)`. `Spacer` import dropped.
  - **DoorHistoryContent error banners**: pre-2.16.30 each `ErrorCard` self-applied `Modifier.padding(top = 16.dp)`. Now the outer Column owns inter-banner spacing via `verticalArrangement = spacedBy(16.dp)`. Top chrome clearance (16 dp below the TopAppBar) is conditionally applied to the outer Column ONLY when at least one banner is shown — without the conditional, the no-banner case would double-pad (16 dp outer + 16 dp `safeListContentPadding` = 32 dp before the first day section). Banner-to-content gap is intentionally 32 dp (16 dp `spacedBy` + 16 dp `safeListContentPadding`) to visually separate the error from the list.
  - **HistoryEmptyState**: pre-2.16.30 the empty-state Column claimed `padding(top = 64.dp)` for itself to push the content visually down. Now the LazyColumn item wraps it in a `Box(modifier = Modifier.fillParentMaxSize(), contentAlignment = Alignment.Center)` so the empty state is **truly vertically centered** in the available height (UX improvement). The Composable itself owns only its internal horizontal padding (`32.dp`) for readable measure; no self-claimed vertical gap.

## 2.16.29
- **Spacing rule: container owns the gap; every screen-level scrollable conforms.** Pre-2.16.29 there was a structural inconsistency between the three "Section"-using pages (Home / Settings / History) and the two bare-list pages (Diagnostics / Function list). Section pages had **16 dp** above the first item (8 dp `safeListContentPadding.top` + 8 dp `SectionHeaderTop` claimed by the section's own Row padding), while the bare-list pages only had **8 dp**. The Status header on Home was visibly higher when at the top than when an alert was rendered above it — the alert claimed only the 8 dp `safeListContentPadding.top`, but Status added the section's own 8 dp on top.
  - **Fix**: `SectionHeaderTop` deleted. The first-item gap is now owned exclusively by the parent `LazyColumn` via `safeListContentPadding.top` (bumped 8 → 16 dp) and `Spacing.ListContentPadding.top` (also 8 → 16 dp). Between-item gap is owned by `Spacing.BetweenItems` (bumped 8 → 16 dp). All five screen-level scrollables now share one rule.
  - **Visible result**: every page has identical 16 dp vertical rhythm — between items AND above the first item, regardless of whether items are `Section`-wrapped or bare. Home's Status header sits at the same y whether it's the first item or whether an alert is above it.
  - **Diagnostics & Function list affected**: the gap between the counters surface and the action button stack on Diagnostics doubles (8 → 16 dp), and rows on Function list double their inter-row gap (12+ rows × +8 dp each). Both screens are dev-facing; the looser rhythm matches the main tabs the user already likes.
  - **Internal: `HomeSection` / `SettingsSection` / `HistoryDaySection` Row + Text padding lost their `top` arg.** The intra-section gap (between the header text and the body card) is now owned by the section's outer `Column` via `verticalArrangement = spacedBy(Spacing.SectionHeaderBottom)` so even that gap follows the parent-owns rule. `SectionHeaderBottom` token unchanged at 8 dp; `SectionHeaderStart` (16 dp horizontal padding inside the header row) unchanged.
  - **Internal: `SpacingTokensPreview` updated** to reflect the new structure.

## 2.13.3
- **Fix attempt: sign-in regression where the UI stays on "Sign in with Google" despite a successful Google account picker flow.** Adds a defensive fallback in `FirebaseAuthRepository`'s reactive listener: when `getIdToken(forceRefresh = false)` returns null (cached path), retry with `getIdToken(forceRefresh = true)` (forced network fetch) before deciding the user is unauthenticated. The previous code's defensive `else` branch wrote `AuthState.Unauthenticated` when the cache was empty — which is exactly what happens in the moment between `signInWithCredential` succeeding and Firebase populating its internal token cache.
- **Why this is "fix attempt" rather than "fix"**: this is a tactical patch built on assumptions that haven't been independently confirmed yet (smoke testing required). Listed assumptions:
  1. The listener firing after a successful sign-in always has a non-null `currentUser`. If `currentUser` is itself null in this race, the patch does not help.
  2. `forceRefresh = true` succeeds when `forceRefresh = false` returns null. If Firebase has the same timing window with forced refresh, the patch does not help.
  3. The fallback adds at most one network round-trip per listener emission. Listener fires once per sign-in / sign-out / explicit refresh — cost is negligible.
  4. Offline launch with previously-signed-in user returns the cached token via the fast path; the forced fallback never runs. No offline regression.
  5. Offline launch with empty cache: both fetches return null, falls through to `Unauthenticated` — same as pre-2.13.3 behavior.
- **Internal: this is a tactical fix in the existing observe-with-side-effect pattern.** The architectural fix is to move token state entirely inside `AuthRepository` (UseCases never see tokens; the listener loop becomes pure observation with no `getIdToken` call).
